# Migrate From Sanchika v0.1.0 To v0.1.1

## Why This Correction Exists

v0.1.1 preserves the v0.1.0 JavaScript, declarations, CSS, exports, and runtime
behavior while pinning the exact release runtime used to create compressed
tarball bytes. Packed manifests and README install text advance to v0.1.1. It
is a GitHub artifact correction, not an npm publication or a new feature
release.

## Installation

The following assets are available only after the v0.1.1 GitHub release is
published:

- `https://github.com/lamemustafa/sanchika/releases/download/v0.1.1/sanchika-tokens-0.1.1.tgz`
- `https://github.com/lamemustafa/sanchika/releases/download/v0.1.1/sanchika-primitives-0.1.1.tgz`
- `https://github.com/lamemustafa/sanchika/releases/download/v0.1.1/sanchika-patterns-0.1.1.tgz`

Declare all three direct tarballs and override the exact internal package
dependencies to the same GitHub assets. The overrides are required because
pnpm otherwise looks up the packed `0.1.1` dependency ranges on npm:

```json
{
  "dependencies": {
    "@sanchika/tokens": "https://github.com/lamemustafa/sanchika/releases/download/v0.1.1/sanchika-tokens-0.1.1.tgz",
    "@sanchika/primitives": "https://github.com/lamemustafa/sanchika/releases/download/v0.1.1/sanchika-primitives-0.1.1.tgz",
    "@sanchika/patterns": "https://github.com/lamemustafa/sanchika/releases/download/v0.1.1/sanchika-patterns-0.1.1.tgz"
  }
}
```

Add the exact internal dependency mappings to the consumer's
`pnpm-workspace.yaml`:

```yaml
overrides:
  "@sanchika/tokens@0.1.1": "https://github.com/lamemustafa/sanchika/releases/download/v0.1.1/sanchika-tokens-0.1.1.tgz"
  "@sanchika/primitives@0.1.1": "https://github.com/lamemustafa/sanchika/releases/download/v0.1.1/sanchika-primitives-0.1.1.tgz"
```

Then install and review the resulting lockfile:

```bash
pnpm install
```

These are GitHub release artifacts, not npm registry packages.

## CSS Order

The required order is unchanged:

```css
@import "@sanchika/tokens/theme.css";
@import "@sanchika/primitives/styles.css";
@import "@sanchika/patterns/styles.css"; /* only when product patterns are used */
@import "./consumer.css"; /* consumer CSS last */
```

## Compatibility Guidance

- Existing v0.0.2 CSS aliases continue to resolve during the documented
  compatibility window; new work should use current semantic names.
- Existing `primitiveClassName` calls for `Button`, `Card`, `Badge`, and `Field`
  retain their valid legacy class output.
- The `PrimitiveSpec` shape and legacy TypeScript token collections remain
  supported.
- Existing legacy pattern exports and aliases remain supported.
- There is no intentional public API, JavaScript, declaration, CSS, export, or
  runtime behavior change from v0.1.0; packed versions, exact internal
  dependency versions, and README install text become v0.1.1.
- `@sanchika/gallery-app` is a private deployed application, not a dependency
  or release package.
- Source package manifests deliberately remain private at version 0.0.0 with
  `workspace:*` dependencies; the pack lane rewrites only clean temporary
  copies.
- Root `release.json` remains the only stable release metadata authority: its
  `version` is the v0.1.1 artifact candidate and its `previousVersion` is the
  published v0.1.0 rollback baseline used by ordinary pre-publication gallery
  builds.
- All APIs remain pre-1.0 and experimental.

## Consumer Verification

Use only public package entrypoints. Run the consumer's typecheck, build,
browser smoke, and package-manager lockfile check after installation. Confirm:

- all three installed package versions are `0.1.1`;
- primitives depends on tokens at exactly `0.1.1`;
- patterns depends on primitives at exactly `0.1.1`;
- token, primitive, and pattern CSS entrypoints resolve in that order;
- no gallery package is installed;
- the three final artifact checksums are recorded in adoption evidence.

## Rollback To v0.1.0

Restore these exact immutable assets together:

- `https://github.com/lamemustafa/sanchika/releases/download/v0.1.0/sanchika-tokens-0.1.0.tgz`
- `https://github.com/lamemustafa/sanchika/releases/download/v0.1.0/sanchika-primitives-0.1.0.tgz`
- `https://github.com/lamemustafa/sanchika/releases/download/v0.1.0/sanchika-patterns-0.1.0.tgz`
- `https://github.com/lamemustafa/sanchika/releases/download/v0.1.0/manifest.json`

1. Replace all three direct dependency URLs and both `pnpm-workspace.yaml`
   override values with the v0.1.0 URLs above.
2. Change the override selectors to `@sanchika/tokens@0.1.0` and
   `@sanchika/primitives@0.1.0`; leaving the v0.1.1 selectors would allow the
   rollback dependencies to fall through to npm.
3. Run `pnpm install` to regenerate the lockfile, review the lockfile diff, and
   confirm all three Sanchika packages resolve to 0.1.0.
4. Run the consumer package-backed smoke, typecheck, build, and browser checks.
5. Commit the consumer manifest and lockfile rollback together.

Rollback affects package references and the lockfile only. It does not require
a database, DNS, or workspace migration. Consumers that must return to the
older compatibility baseline can follow
[the v0.0.2 rollback](./v0.0.2-to-v0.1.0.md#rollback-to-v002) after first
removing v0.1-only API and CSS usage.
